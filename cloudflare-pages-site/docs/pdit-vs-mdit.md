# PDIT vs MDIT

这份文档用来快速区分当前仓库里的两条研究线。

## PDIT

- 输入模态：点云为主
- 目标函数：Flow Matching / Diffusion
- 条件形式：`obs token -> trajectory backbone`
- 当前状态：已验证稳定，当前金标准
- 当前 best：`100 episodes = 0.85`

## MDIT

- 输入模态：RGB + text + state
- 目标函数：以 `PDIT` 主线为骨干语义的 Flow Matching
- 条件形式：`5 路 CLIP 视觉特征 + 文本投影 + 状态适配 -> step fusion -> 3 cond tokens`
- 当前状态：共享 `PDIT` 骨干、动作后处理和审计链的 RGB+文本主线
- 当前 best：`20 episodes = 0.75 @ epoch 300/500`

## 为什么要并列而不是兼容

原因不是“它们完全无关”，而是“当前仓库要把输入侧变量和骨干侧变量拆开看”：

- `pdit` 已经是稳定强基线，不应该为了给 RGB+文本让路而改坏点云主线
- 当前成功的 `mdit` 不是原版 `MTDP` 的直接复现，而是“保留 `PDIT` 骨干和控制语义，只把输入换成 `5RGB + text`”
- 如果把“输入替换”和“骨干重写”混在一起，最后很难判断成功到底来自哪里

所以现在的原则是：

- `pdit` 继续做稳定对照组和行为锚点
- `mdit` 在共享骨干和共享评估链下验证“点云替换成 RGB+文本是否能稳定成立”
- `mdit` 当前最关键的差异集中在输入侧，而不是另起一套完全独立的方法学

## 当前应如何理解两条线

可以把当前仓库里的关系理解成下面两句：

- `PDIT`：已经验证成功的点云主线，提供稳定的骨干、控制语义和审计口径。
- `MDIT`：在尽量不动上述稳定部分的前提下，把输入升级为 `5RGB + text + robot_state` 的研究线。

因此，当前 `MDIT` 成功的关键并不是“faithful MTDP 原样复现成功”，而是：

1. 只替换输入侧，不同时改坏骨干和执行链。
2. RGB+文本条件通过分阶段适配压成 `3 cond tokens`，继续沿用 `PDIT` 的条件使用方式。
3. Flow Matching、动作后处理和共享 audit 链都保留现有稳定工程语义。
